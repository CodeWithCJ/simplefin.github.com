SimpleFIN

Introduction

SimpleFIN (also SFIN) stands for Simple Financial Interchange and aims to make read-only financial interchange simple. Think of it like RSS for financial information.

The SimpleFIN standard defines:

  1. SimpleFIN data format
  2. SimpleFIN HTTP protocol, including authentication

You should use SimpleFIN because:

  • All access is read-only.
  • You can read and comprehend the spec in a single day.
  • You don't have to pay anyone to use it (ahem, QFX).

SimpleFIN data may be obtained either directly from a financial institution or from a third party that previously obtained the original data from a financial institution. In other words, both banks and financial aggregation services may provide SimpleFIN data and sevices.

HTTP Protocol
  1. HTTP servers must require SSL for all SimpleFIN communication.
  2. As noted in Organization's sfin-url attribute, an SimpleFIN HTTP server must designate a root URL as the root SimpleFIN resource. For instance, https://sfin.mybank.com or https://www.mybank.com/sfin. All paths noted below are relative to this root URL.
Servers must provide the following paths:

Method Path Response
GET /accounts An Account Set for all accounts accessible to the client from this server. See /accounts for more information.
GET /accounts/:id An Account Set for a specific Account. See /accounts/:id for more information.

/accounts

 
GET /accounts

Fetch a list of all accounts available to the requester.

  1. If no query parameters are supplied, no Account.transactions will be returned.

Example response body

{ "accounts": [ { "org": { "domain": "mybank.com", "sfin-url": "https://sfin.mybank.com/" }, "id": "2930002", "name": "Savings", "currency": "USD", "balance": "100.23", "available-balance": "75.23", "last-transaction-id": "AO334", } ] ]

Optional query parameters

Name Example Description
start-date ?start-date=2001-05-15 If given, transactions on or after the supplied date or timestamp will be returned.
end-date ?end-date=2001-06-19 If given, only transactions before the supplied date or timestamp will be returned. Transactions with a posted date equal to this date/timestamp will not be returned. For instance, querying for ?start-date=2001-05-01&end-date=2001-06-01 will return all transactions from May, 2001.

/accounts/:id

 
GET /accounts/:id

Fetch a list of the most recent transactions for a particular account (specified by the :id).

For instance, if the response from GET /accounts indicates that there's an account with id of 1392, the transactions for that account will be found at /accounts/1392.

  1. If no domain query parameter is supplied, only accounts originating from the same domain as the SimpleFIN server will be returned.

    For example, 
    https://www.mybank.com/sfin/accounts/10
    corresponds to account 10 originating from mybank.com.

    But 
    https://www.mybank.com/sfin/accounts/10?domain=foo.com
    corresponds to account 10 originating from foo.com.
  2. Servers must respond with all transactions for at least the past 60 days, though more than 60 days of transactions may be returned.
  3. Transactions must be sorted by id, with the latest transaction last.
  4. The last-transaction-id must be the last Transaction listed;
  5. The balance and available-balance must be the balance resulting from the last transaction listed.

Example response body

{ "accounts": [ { "org": { "domain": "mybank.com", "sfin-url": "https://sfin.mybank.com/" }, "id": "2930002", "name": "Savings", "currency": "USD", "balance": "100.23", "available-balance": "75.23", "last-transaction-id": "AO335", "transactions": [ { "id": "AO334", "posted": "2001-01-01T09:22:33.2343", "amount": "-22.33", "memo": "John's Pizza Pie Castle, Cafe and Spa", "sic": "2354" }, { "id": "AO335", "posted": "2001-01-05T10:33:29.82", "amount": "-5.02", "memo": "Frodo's Ring Repair", "sic": "9003" } ] } ] }

Response code Applicable paths Scenario
403 Forbidden /accounts
/accounts/:id		 If the requester is not authenticated or an invalid token is used.
404 Not Found /accounts/:id If the requested account does not exist.
404 Not Found /accounts/:id If the requested account exists, but the requester is not authorized to view it. 404 is used instead of 403 to deter brute-force account listing.

Authentication

 HTTP Bearer Authentication is used. In every request, the client sends an Authentication header of the format: 
Authentication: Bearer TOKEN
where TOKEN is replaced with a valid token received from the operator of the SimpleFIN server. The format and length of tokens is determined by the SimpleFIN server operator, keeping the following in mind:
  1. Tokens must be alphanumeric
  2. Tokens should be randomly generated, rather than being based upon potentially sensitive account information.

Acquiring tokens

The process for acquiring tokens is not part of the SimpleFIN standard. An organization operating an SimpleFIN server may do whatever they'd like to deliver tokens to users, including sending mail or email, providing a web interface for managing tokens, or using other established processes such as OAuth or OpenID.

Users must be allowed to distribute tokens to third parties.

Recommendations

This standard recommends that, whatever process is used, users have the following abilities:

  1. Users can associate user-provided names with tokens.
  2. Users can create many tokens (for providing to third party services).
  3. Users can choose an expiration date for tokens when creating them.
  4. Users can manually invalidate/expire existing tokens at will. This is useful in the case of stolen tokens or abuse by third parties to whom tokens were given.

Additionally, consider the following when implementing a token management system:

  1. It may be nice to notify users prior to token expiration.
  2. It may be nice to allow users to temporarily disable all tokens and later re-enable them. This can be used to prevent access while doing research into abuse.
  3. Consider providing logging/audit information regarding token use, including the time tokens were used and the IPs from which requests came.
  4. Allow tokens to be associated with a subset of available accounts. For instance, allow the user to create one token for access to just their checking account, and another token for access to just their savings account.
